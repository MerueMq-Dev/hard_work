# Итерация 1: Три одинаковых метода PreCalc/Calc/ReCalc

## Исходный код (без комментариев уровня 3)

```csharp
internal async Task PreCalc(Calculation calculation)
{
    calculation.ClearDiscountAndPromotions();
    calculation.InitDiscountedDishes();
    
    if (calculation.AntifraudBlocked)
        return;
    
    await DiscountPromotionPromocodeBonusFlow(calculation);
    
    // начисляем Бонусы на сумму которая осталась после всех Скидок,Акций,Промокодов и списания Бонусов как скидки
    AccrueBonuses(calculation);
}

internal async Task Calc(Calculation calculation)
{
    calculation.ClearDiscountAndPromotions();
    calculation.InitDiscountedDishes();
    
    if (calculation.AntifraudBlocked)
        return;
    
    await DiscountPromotionPromocodeBonusFlow(calculation);
    
    // начисляем Бонусы на сумму которая осталась после всех Скидок,Акций,Промокодов и списания Бонусов как скидки
    AccrueBonuses(calculation);
}

internal async Task ReCalc(Calculation calculation)
{
    calculation.ClearDiscountAndPromotions();
    calculation.InitDiscountedDishes();
    
    if (calculation.AntifraudBlocked)
        return;
    
    await DiscountPromotionPromocodeBonusFlow(calculation);
    
    // начисляем Бонусы на сумму которая осталась после всех Скидок,Акций,Промокодов и списания Бонусов как скидки
    AccrueBonuses(calculation);
}
```

## Какие вопросы возникают при чтении?

Вижу три идентичных метода. Единственный комментарий описывает что делается (начисляем бонусы), но не почему есть три копии одного кода.

Сразу возникают вопросы. В чём разница между *PreCalc*, *Calc* и *ReCalc*? Код одинаковый, имена намекают на "до", "во время" и "после", но конкретики нет. Когда какой метод вызывать? Из кода это не понятно. Почему три копии, а не один метод? Может планировалось что они будут отличаться? Безопасно ли их объединить? Есть ли тонкая разница в поведении? Что такое "антифрод блокировка"? Почему при ней мы просто выходим?

Имя метода *PreCalc* говорит что (предварительный расчёт), но не говорит зачем он нужен и когда его использовать вместо *Calc*.

## Скрытая информация уровня 3 (что держал в голове автор)

Порывшись в коде дальше и спросив у автора, выясняю контекст использования. 

*PreCalc* вызывается когда клиент добавляет блюда в корзину на сайте. Показывает "сколько будет стоить с учётом скидок", но не сохраняет расчёт в базу. 

*Calc* вызывается когда клиент подтверждает заказ. Сохраняет расчёт в базу и резервирует промокод/бонусы. 

*ReCalc* вызывается когда клиент изменяет состав заказа уже после создания. Пересчитывает всё заново и обновляет запись в базе.

По бизнес-логике все три метода делают одинаковый расчёт, разница только в побочных эффектах: сохранение или обновление. Разделение на три метода было временным решением на старте проекта, потом забыли объединить. *AntifraudBlocked* означает что клиент попал в чёрный список. Ему нельзя давать скидки/бонусы, только базовую цену.

Есть важные инварианты. Порядок применения строгий: сначала скидки, потом акции, потом промокоды, потом списание бонусов, в конце начисление новых бонусов. Начисление бонусов идёт на сумму после всех скидок, иначе можно накрутить бонусы. Если антифрод заблокировал — нельзя начислять бонусы вообще, даже на полную сумму, это политика компании.

В TODO лежит объединить три метода в один с параметром *mode* со значениями Preview/Create/Update, и вынести сохранение в базу из сервиса расчёта, это нарушение SRP.

Вся эта информация критична для понимания кода, но ни слова из неё нет в исходном коде!

## Код с правильными комментариями уровня 3

```csharp
/// 
/// Сервис расчёта скидок/акций/промокодов/бонусов.
/// Порядок важен: Скидки -> Акции -> Промокоды -> Списание бонусов -> Начисление бонусов.
/// Начисление только после всех скидок (инвариант: иначе можно накрутить).
/// TODO: Объединить PreCalc/Calc/ReCalc в один Calculate(calculation, mode).
/// 
public class CalculationService
{
    /// 
    /// Preview: показать цену в корзине. Не сохраняет, не резервирует промокод.
    /// 
    internal async Task PreCalc(Calculation calculation)
    {
        calculation.ClearDiscountAndPromotions();
        calculation.InitDiscountedDishes();
        
        // Антифрод: заблокированным только базовая цена, без скидок/бонусов
        if (calculation.AntifraudBlocked)
            return;
        
        await DiscountPromotionPromocodeBonusFlow(calculation);
        AccrueBonuses(calculation);
    }

    /// 
    /// Create: подтверждение заказа. Сохраняет в базу, резервирует промокод.
    /// TODO: Убрать сохранение отсюда (нарушение SRP).
    /// 
    internal async Task Calc(Calculation calculation)
    {
        calculation.ClearDiscountAndPromotions();
        calculation.InitDiscountedDishes();
        
        if (calculation.AntifraudBlocked)
            return;
        
        await DiscountPromotionPromocodeBonusFlow(calculation);
        AccrueBonuses(calculation);
    }

    /// 
    /// Update: клиент изменил состав заказа. Пересчёт + обновление в базе.
    /// 
    internal async Task ReCalc(Calculation calculation)
    {
        calculation.ClearDiscountAndPromotions();
        calculation.InitDiscountedDishes();
        
        if (calculation.AntifraudBlocked)
            return;
        
        await DiscountPromotionPromocodeBonusFlow(calculation);
        AccrueBonuses(calculation);
    }

    /// 
    /// Почему после скидок: иначе накрутят бонусы, купив со скидкой 90%
    /// но получив бонусы как за полную цену.
    /// 
    private void AccrueBonuses(Calculation calculation)
    {
        calculation.DepositBonuses.Reset();
        Calculator.CalculateBonus(calculation);
    }
}
```

## Выводы

Контекст использования теперь явно документирован в комментариях к каждому методу. Видно когда вызывать *PreCalc* против *Calc* против *ReCalc*. Побочные эффекты тоже задокументированы явно, из кода не видно что *Calc* сохраняет в базу (это скрыто внутри вызовов), комментарий делает это явным.

Бизнес-правила вынесены в комментарий к классу: порядок применения, инвариант про начисление бонусов. TODO задокументированы: объединить три метода, убрать сохранение из сервиса. Эта информация критична для будущего рефакторинга, но без комментария её нет.

Комментарий к *AccrueBonuses* объясняет почему начисление идёт после скидок (иначе можно накрутить). Из кода это не видно.

Информация которая не может быть закодирована в коде: когда вызывать какой метод (решается на уровне API-контроллера), побочные эффекты (сохранение в базу скрыто внутри других вызовов), бизнес-причины решений (почему начисление после скидок), TODO и планы на будущее, политики компании (антифрод блокирует бонусы).

Все попытки "закодировать" эту информацию в имена методов привели бы к монстрам типа *PreCalcWithoutSavingForCartPreview*, это абсурд. Комментарии единственный способ сохранить контекст уровня 3.

# Итерация 2: Цепочка проверок лимита скидки

## Исходный код (с комментариями уровня 2)

```csharp
private async Task DiscountPromotionPromocodeBonusFlow(Calculation calculation)
{
    /// если скидку нельзя применить, достигли потолка Скидки - выходим
    if (calculation.TotalLimitDiscount <= calculation.TotalDiscount)
        return;

    var bestDiscount = await _repository.FindBestDiscount(
        calculation.CorporationId,
        calculation.CreateDate, 
        DateTimeOffset.UtcNow, 
        calculation.GuestPhone);

    Calculator.CalculateDiscount(calculation, bestDiscount, DiscountTypeEnum.discount);


    /// если скидку нельзя применить, достигли потолка Скидки - выходим
    if (calculation.TotalLimitDiscount <= calculation.TotalDiscount)
        return;

    var actualPromos = await _repository.FindActivePromotions(
        calculation.CorporationId,
        calculation.CreateDate, 
        DateTimeOffset.UtcNow);

    await Calculator.CalculatePromotions(calculation, actualPromos);

    /// если скидку нельзя применить, достигли потолка Скидки - выходим
    if (calculation.TotalLimitDiscount <= calculation.TotalDiscount)
        return;

    /// считаем сколько Бонусов можем в теории списать
    await Calculator.ExpectWithdrawBonus(calculation);
    
    await Calculator.CheckAndWithdrawBonus(calculation);
}
```

## Какие вопросы возникают при чтении?

Комментарии есть, но они описывают что делается ("если достигли потолка - выходим"), а не почему это важно.

Вопросы сразу возникают. Почему проверка повторяется три раза? Код кажется избыточным. Что такое *TotalLimitDiscount*? Это какой-то глобальный лимит на скидки? Почему проверка между шагами, а не в конце? Зачем проверять после каждого шага? Что будет если убрать промежуточные проверки? Код сломается или просто применит больше скидок? Почему нет проверки перед *ExpectWithdrawBonus*? Для скидок/акций есть, а для бонусов нет. Можно ли изменить порядок? Например, сначала акции, потом скидки?

Имена переменных понятны (*bestDiscount*, *actualPromos*), но контекст зачем нужны эти проверки не виден.

## Скрытая информация уровня 3 (что держал в голове автор)

Порывшись в требованиях и спросив у бизнеса, выясняю бизнес-правило про лимит совокупной скидки.

*TotalLimitDiscount* это максимальная скидка которую можно дать клиенту на заказ. Устанавливается администратором ресторана, например 30% или 50%. Почему это нужно? Без лимита клиент может собрать купон на 20%, акцию на 15%, промокод на 10% и получить заказ за полцены или вообще бесплатно. Ресторан теряет деньги. Инвариант такой: сумма всех скидок (discount плюс promotions плюс promocode плюс bonus) не должна превышать *TotalLimitDiscount*.

По поводу жадного алгоритма. Проверка после каждого шага это не избыточность, а жадный алгоритм: "применяй скидки пока не упрёшься в лимит". Если после применения скидок лимит исчерпан, нет смысла даже искать акции, экономим запрос к базе.

Порядок выбран так чтобы максимизировать выгоду для клиента в рамках лимита. Сначала персональные скидки (для VIP клиентов), потом акции (могут быть комбинируемые), потом промокоды, в конце бонусы.

Почему именно такой порядок? Персональные скидки приоритетнее потому что они даны клиенту вручную менеджером, их нельзя игнорировать. Акции идут следующими потому что они комбинируемые (можно применить несколько одновременно), и они выгоднее промокодов. Промокоды идут после акций потому что промокод одноразовый, а акции постоянные. Бонусы в конце потому что это "деньги клиента", если он хочет их потратить, мы должны дать ему эту возможность, но только после всех маркетинговых инструментов.

Edge case с бонусами. Перед *ExpectWithdrawBonus* нет проверки лимита намеренно. Почему? Функция *ExpectWithdrawBonus* сама учитывает лимит внутри себя и рассчитывает сколько бонусов можно списать в рамках оставшегося лимита. Если лимит исчерпан, *ExpectWithdrawBonus* установит статус "noaction" и бонусы не спишутся.

Историческая причина тоже есть. Изначально был только один тип скидок, лимита не было. Потом добавили акции и промокоды, возникла проблема "слишком большая скидка". Ввели *TotalLimitDiscount*, пришлось добавить проверки между шагами. Рефакторинг не делали потому что "работает, не трогай".

Вся эта информация отсутствует в комментариях!

/// Проверка после каждого шага экономит запросы к базе.
/// Порядок: 1.Персональные скидки 2.Акции 3.Промокоды 4.Бонусы (не менять!).

## Выводы

Контекст бизнес-правила теперь явно документирован: что такое *TotalLimitDiscount*, почему он нужен, что будет без него. Обоснование алгоритма тоже есть, почему проверка после каждого шага (жадный алгоритм плюс оптимизация), а не один раз в конце.

Обоснование порядка явное: почему именно такая последовательность, что будет если изменить. Есть явное предупреждение "не менять". Edge case с бонусами объяснён, почему перед бонусами нет проверки лимита. Без комментария это выглядит как баг или забывчивость.

## Общая рефлексия

Работая над этим заданием и примерами из проекта, я довольно чётко осознал одну вещь, которую раньше скорее знал теоретически, но не проживал на практике: код по своей природе ограничен уровнем реализации и не способен передать дизайн, контекст и причины решений. Всё, что относится к уровню 3, живёт исключительно в голове разработчика — пока он это явно не зафиксирует.

В обеих итерациях я столкнулся с одной и той же проблемой, просто в разных формах.

В первой итерации с PreCalc / Calc / ReCalc код выглядел как очевидная копипаста, которую «руки чешутся» объединить. И если смотреть только на реализацию, это действительно выглядит как плохой код. Но как только появляется контекст — preview корзины, создание заказа, пересчёт уже сохранённого — становится ясно, что это не просто дублирование, а отражение разных сценариев использования. Без комментариев я этого контекста не вижу, и именно это делает код опасным для рефакторинга.

Во второй итерации с проверками лимита скидки ситуация повторилась. Код показывает, что мы делаем (проверяем лимит и выходим), но полностью скрывает почему это сделано именно так. Без комментариев проверки выглядят избыточными, порядок — случайным, а отсутствие проверки перед бонусами — багом. Только когда я сформулировал бизнес-правила и мотивацию алгоритма, стало понятно, что это не «грязный код», а жадный алгоритм с осознанными оптимизациями. Эта информация принципиально не читается из кода.

Из-за этого я по-другому посмотрел на фразу из материала о том, что «фактический вызов API никак не связан с дизайном». В моём случае интеграция с Telegram ботом полностью теряется, если смотреть только на enum и методы. Система есть, но в коде она не проявлена как система — пока я не описал её словами.

Отдельно для себя я понял, почему многие разработчики не любят комментарии. Хороший комментарий уровня 3 — это не пересказ кода, а попытка поговорить с будущим читателем: объяснить мотивацию, ограничения, последствия неправильных изменений. Это действительно сложнее, чем написать код. Но именно это и делает комментарий ценным.

Раньше мои комментарии в основном были уровня 2: «начисляем бонусы», «проверяем лимит». Теперь я ясно вижу, что такие комментарии почти бесполезны. Они не добавляют информации, потому что дублируют код. Ценность появляется только там, где я объясняю почему именно так, зачем это нужно системе и что нельзя делать, даже если кажется логичным.

Ещё один важный инсайт — роль TODO. Во всех трёх примерах самые критичные знания о будущем кода существовали только в виде комментариев: «объединить методы», «убрать Flags», «добавить таймаут». Эти вещи невозможно выразить в коде вообще. Код либо выполняется, либо нет. Он не может сказать: «это технический долг» или «это компромиссное решение». Если это не зафиксировано комментариями — это знание потеряется.

В итоге я пришёл к выводу, что комментарии уровня 3 — это не признак слабого кода и не «подпорка для непонятной реализации». Это форма уважения к другим разработчикам и к себе в будущем. Потратив несколько минут на объяснение контекста и причин, я экономлю часы чужого времени и снижаю риск неправильных изменений.

Для себя я сформулировал простой принцип, который хочу применять дальше:
перед тем как писать или менять код, стоит задать себе вопросы «зачем это существует?», «в каком месте системы это используется?» и «что нельзя делать с этим кодом». Если ответы не очевидны из сигнатур и структуры — они должны быть записаны в комментариях.

Код может быть сколько угодно «чистым» на уровне реализации, но без комментариев уровня 3 он остаётся немым. Комментарии — единственный способ сохранить и передать знания о дизайне, контексте и причинах решений.
