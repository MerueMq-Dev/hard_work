# Отчёт: Примеси (Mixins) в C#

## Введение

Честно говоря, когда увидел задание про примеси и понял, что это просто интерфейсы — первая реакция была «ну, это же просто интерфейсы». Взял три класса из проекта — *Promotion*, *Promocode*, *GuestSegment* — и начал разбираться. Оказалось, что разница между «просто интерфейсом» и «примесью» не синтаксическая, а смысловая. И проявляется она именно на живом коде, а не на абстрактных примерах из статей.

Главная мысль, которую вынес: примесь — это способ дать классу готовое поведение без наследования. В C# это реализуется через **Default Interface Methods**: метод с телом прямо в интерфейсе. Класс, реализующий интерфейс, получает этот метод бесплатно. Хочет — переопределяет. Не хочет — использует как есть.

Разница с наследованием критическая: от одного базового класса можно унаследоваться только один раз. Интерфейсов можно реализовать сколько угодно. Это значит, что каждый класс берёт ровно те «способности», которые ему нужны — не больше.

Ещё по ходу работы понял, что примеси бывают двух видов, и их важно не путать: **доменные** — несут бизнес-правило (*IAuditable*, *IDiscountCalculable*), и **инфраструктурные** — отвечают за технические аспекты (*ILoggable*). Они живут в разных слоях, и смешивать их — значит тащить инфраструктуру в домен. Это разграничение всплывало в каждой итерации.

## Итерация 1: Дублирование поведения — код без примесей

### Исходный код (без примесей)

Три класса домена. Каждый умеет логировать и запрашивать аудит. Но реализация — своя в каждом:

```csharp
public class Promotion
{
    public string Name { get; }
    public decimal DiscountPercent { get; }
    public string? ActivatedBy { get; }

    // Логирование — своя реализация
    public void Log(string message)
    {
        var ts = DateTime.UtcNow.ToString("HH:mm:ss");
        Console.WriteLine($"[{ts}] [Promotion] {message}");
    }

    // Аудит — своя реализация
    public string GetAuditInfo() =>
        ActivatedBy is null
            ? "Активировано автоматически"
            : $"Активировано пользователем: {ActivatedBy}";
}

public class Promocode
{
    public string Code { get; }

    // Логирование — точная копия из Promotion
    public void Log(string message)
    {
        var ts = DateTime.UtcNow.ToString("HH:mm:ss");
        Console.WriteLine($"[{ts}] [Promocode] {message}");
    }
}

public class GuestSegment
{
    public string Name { get; }
    public string? ActivatedBy { get; }

    // Снова логирование, снова аудит
    public void Log(string message) { ... }
    public string GetAuditInfo() => ...
}
```

### Какие вопросы возникают при чтении?

Код работает, но сразу бросается в глаза одна вещь — метод *Log* в каждом классе идентичен по смыслу, но написан заново. Возникают конкретные вопросы.

- Почему *Log* продублирован три раза? Это намеренно или забыли вынести?
- Если логика изменится (например, добавить название сервиса), нужно менять в трёх местах?
- *GetAuditInfo* в *Promotion* и *GuestSegment* — это одно и то же правило? Или они могут разойтись?
- Как понять, что *Promocode* принципиально не нужен аудит, а не просто забыли добавить?

Код на уровне реализации понятен. Но отсутствует информация о намерении: это случайное дублирование или осознанное решение держать их раздельно?

### Скрытая информация (что держал в голове автор)

Логика *Log* и *GetAuditInfo* одинакова для всех трёх классов — это общее поведение системы, не специфика отдельного класса. Дублирование появилось потому что классы создавались в разное время и разными итерациями. Каждый раз копировали то, что уже было — и никто не останавливался, чтобы вынести общее.

Намерение было хорошим: если захотим изменить формат лога — меняем в одном месте. Реальность оказалась другой: меняем в трёх. Ещё один важный момент: *Promocode* намеренно не имеет аудита — ему не нужно хранить «кто активировал», это одноразовый код. Но из кода это абсолютно не видно. Читается как пропуск, а не как решение.

### Код с примесями

Выносим поведение в интерфейсы с реализацией по умолчанию. Каждый класс берёт только то, что нужно.

Сразу важная оговорка про *ILoggable*: это инфраструктурная примесь. *Console.WriteLine* в теле интерфейса — это жёсткая зависимость от конкретного вывода прямо в доменном коде. В реальном проекте *ILoggable* должен задавать только контракт *void Log(string message)* без тела, а реализация уходит через *ILogger* из DI в инфраструктурный слой. В примере ниже *Console.WriteLine* оставлен для наглядности, но в *Loyalty.Domain* так делать не стоит.

```csharp
// Примесь 1: логирование — инфраструктурная примесь
// В реальном коде: только контракт без тела, реализация через ILogger в Infrastructure
public interface ILoggable
{
    void Log(string message)
    {
        var ts = DateTime.UtcNow.ToString("HH:mm:ss");
        Console.WriteLine($"[{ts}] [{GetType().Name}] {message}");
    }
}

// Примесь 2: аудит — доменная примесь, живёт в Loyalty.Domain
// Свойство ActivatedBy — обязательный контракт: класс должен его предоставить
public interface IAuditable
{
    string? ActivatedBy { get; }  // не магия — класс обязан реализовать это свойство

    string GetAuditInfo() =>
        ActivatedBy is null
            ? "Активировано автоматически"
            : $"Активировано пользователем: {ActivatedBy}";
}

// Promotion берёт обе примеси
public class Promotion : ILoggable, IAuditable
{
    public string? ActivatedBy { get; }  // выполняем контракт IAuditable
    // Log() и GetAuditInfo() — уже есть, ничего писать не нужно
}

// Promocode берёт только логирование — аудит не нужен
public class Promocode : ILoggable
{
    // Log() — уже есть
    // IAuditable здесь нет — это намеренное решение, теперь видно из сигнатуры
}

// GuestSegment берёт обе
public class GuestSegment : ILoggable, IAuditable
{
    public string? ActivatedBy { get; }  // выполняем контракт IAuditable
}
```

### Выводы

Дублирование исчезло физически. Изменить формат лога теперь нужно в одном месте — в интерфейсе *ILoggable*. Намерение стало видимым: *Promocode* не реализует *IAuditable* — это теперь явное архитектурное решение, а не забытый метод. Именно это меня больше всего зацепило: примесь не просто убирает копипасту, она делает намерения видимыми в сигнатуре класса.

Ещё один момент, который поначалу не был очевиден: *ActivatedBy { get; }* в интерфейсе *IAuditable* — это не магия и не автоматическая реализация. Это обязательный контракт. Компилятор потребует, чтобы каждый класс, реализующий *IAuditable*, сам предоставил это свойство. Метод *GetAuditInfo* в теле интерфейса может на него опереться, потому что знает: оно точно будет. Именно так примесь отличается от простого хелпера — она опирается на гарантии контракта.

## Итерация 2: Бизнес-логика расчёта скидки в примеси

### Исходный код (без примеси)

Логика расчёта скидки нужна и *Promotion*, и *Promocode*. Без примеси — снова копипаста:

```csharp
public class Promotion
{
    public decimal DiscountPercent { get; }

    // Бизнес-логика прямо в классе
    public decimal ApplyDiscount(decimal amount) =>
        amount * (1 - DiscountPercent / 100m);

    public decimal CalculateSaving(decimal amount) =>
        amount - ApplyDiscount(amount);
}

public class Promocode
{
    public decimal DiscountPercent { get; }

    // Точная копия из Promotion
    public decimal ApplyDiscount(decimal amount) =>
        amount * (1 - DiscountPercent / 100m);

    public decimal CalculateSaving(decimal amount) =>
        amount - ApplyDiscount(amount);
}
```

### Какие вопросы возникают при чтении?

- Почему формула одинаковая? Это намеренно или так совпало?
- Если изменятся правила расчёта (например, добавится НДС), где ещё нужно менять?
- Что означает *DiscountPercent <= 0 or > 100* — это валидация или бизнес-правило? Кто за это отвечает?
- Можно ли работать с *Promotion* и *Promocode* через один тип, если логика одинакова?

### Скрытая информация (что держал в голове автор)

Формула расчёта скидки — единое бизнес-правило системы. Она одинакова для всех инструментов: скидок, акций, промокодов. Инвариант: *DiscountPercent* должен быть в диапазоне *(0, 100]*, иначе расчёт теряет смысл.

Были планы добавить метод *ApplyDiscount* в интерфейс *IDiscountCalculable*, чтобы работать с любым инструментом скидок через один тип — для полиморфного применения в *DiscountPromotionPromocodeBonusFlow*. Без этого приходится делать проверки типов или дублировать вызовы.

### Код с примесью

```csharp
// Примесь: алгоритм расчёта скидки — одно место для всей системы
// Доменная примесь, живёт в Loyalty.Domain
//
// ⚠️ Важная оговорка: фиксировать формулу расчёта прямо в интерфейсе —
// не лучшая практика для продакшена. Если появятся инструменты с другой
// моделью скидки (например, фиксированная сумма вместо процента), интерфейс
// придётся ломать или городить обходные решения. Здесь это показано как
// учебный пример примеси с бизнес-логикой, а не как рекомендуемый паттерн.
public interface IDiscountCalculable
{
    // Обязательный контракт — каждый класс должен предоставить это свойство
    decimal DiscountPercent { get; }

    // Инвариант: DiscountPercent должен быть в (0, 100]
    // Проверка централизована здесь — не размазана по каждому классу
    decimal ApplyDiscount(decimal amount)
    {
        if (DiscountPercent is <= 0 or > 100)
            throw new InvalidOperationException(
                $"Некорректный процент скидки: {DiscountPercent}");

        return amount * (1 - DiscountPercent / 100m);
    }

    decimal CalculateSaving(decimal amount) =>
        amount - ApplyDiscount(amount);
}

// Promotion получает расчёт бесплатно
public class Promotion : ILoggable, IAuditable, IDiscountCalculable
{
    public decimal DiscountPercent { get; }  // выполняем контракт IDiscountCalculable
    // ApplyDiscount и CalculateSaving — уже есть из примеси
}

// Promocode тоже
public class Promocode : ILoggable, IDiscountCalculable
{
    public decimal DiscountPercent { get; }  // выполняем контракт IDiscountCalculable
}
```

### Выводы

Инвариант теперь живёт в одном месте — в интерфейсе. Если бизнес-правило изменится, меняем один метод, а не ищем все места где дублировалась формула. Плюс теперь можно работать через тип *IDiscountCalculable* — это открывает путь к полиморфному применению скидок в основном flow расчёта.

Оговорка из кода выше заслуживает отдельного внимания. *IDiscountCalculable* с жёстко зафиксированной формулой хорошо работает пока скидка — это всегда процент. Как только появится инструмент с другой моделью (фиксированная сумма, ступенчатая скидка), интерфейс станет ограничением. Правильнее была бы абстракция *IDiscount* без реализации и разные классы-стратегии под каждую модель. Но это уже паттерн «Стратегия», а не примесь — и выбор между ними зависит от того, насколько правило стабильно в конкретном проекте.

## Итерация 3: Полиморфизм через примесь

### Проблема

В *DiscountPromotionPromocodeBonusFlow* есть последовательность применения инструментов скидок. Без общего интерфейса каждый инструмент вызывается отдельно:

```csharp
// Без полиморфизма — каждый инструмент вызывается вручную
private void ApplyAllDiscounts(Calculation calculation, decimal orderAmount)
{
    // Promotion
    if (promotion != null)
    {
        var saving1 = promotion.CalculateSaving(orderAmount);
        promotion.Log($"Применена акция: экономия {saving1}");
    }

    // Promocode — почти то же самое
    if (promocode != null)
    {
        var saving2 = promocode.CalculateSaving(orderAmount);
        promocode.Log($"Применён промокод: экономия {saving2}");
    }

    // Если добавится новый инструмент — снова дублируем блок
}
```

### Что меняет примесь — полиморфный вызов

Когда оба класса реализуют *IDiscountCalculable*, можно работать через коллекцию интерфейсов, и добавление нового инструмента не требует изменения самого метода.

Но здесь я сразу попался на соблазн написать *if (instrument is ILoggable loggable)* — и это оказался запах. Проверка типа внутри бизнес-логики говорит о том, что абстракция неполная. Если логирование нужно всем инструментам в этом потоке, оно должно быть частью *IDiscountCalculable* напрямую, а не угадываться через *is*. Или решаться декоратором снаружи — тогда бизнес-метод вообще не знает про логирование.

```csharp
// ❌ Так писать не стоит — проверка типа в бизнес-логике
foreach (var instrument in instruments)
{
    var saving = instrument.CalculateSaving(orderAmount);
    if (instrument is ILoggable loggable)  // запах: абстракция неполная
        loggable.Log($"экономия {saving:C}");
}

// ✅ Если логирование нужно всем — включаем его в контракт явно
public interface IDiscountCalculable
{
    decimal DiscountPercent { get; }
    decimal ApplyDiscount(decimal amount);
    decimal CalculateSaving(decimal amount);
    void Log(string message);  // теперь полиморфизм явный и чистый
}

// Тогда метод выглядит так — никаких проверок типов
private void ApplyAllDiscounts(
    IEnumerable<IDiscountCalculable> instruments,
    decimal orderAmount)
{
    foreach (var instrument in instruments)
    {
        var saving = instrument.CalculateSaving(orderAmount);
        instrument.Log($"{instrument.GetType().Name}: экономия {saving:C}");
    }
}

// Использование: передаём любые объекты с IDiscountCalculable
IDiscountCalculable[] instruments = { promotion, promocode };
ApplyAllDiscounts(instruments, orderAmount);

// Добавили новый тип — просто добавляем в массив, метод не трогаем
IDiscountCalculable[] instruments = { discount, promotion, promocode };
```

### Какие вопросы возникают без примеси?

- Откуда вызывается каждый блок? Порядок важен, но он нигде не зафиксирован явно.
- Как добавить новый инструмент скидок, не сломав существующий поток?
- Почему один блок обрабатывается иначе? Это намеренно или ошибка при копипасте?

### Выводы

Примесь здесь работает как контракт и реализация одновременно. Метод *ApplyAllDiscounts* не знает, что именно применяет — *Promotion*, *Promocode* или что-то новое. Он знает только, что у объекта есть *CalculateSaving* и *Log*. Это и есть полиморфизм через примесь — явный, без проверок типов внутри бизнес-кода.

Третья итерация показала важное правило: если в цикле появляется *is SomeInterface* — это сигнал пересмотреть контракт. Либо включить нужное поведение в основной интерфейс явно, либо вынести логирование в декоратор. Но не оставлять проверку типа в бизнес-методе — это дыра в абстракции.

> **Связь с проектом:** в *DiscountPromotionPromocodeBonusFlow* сейчас каждый инструмент вызывается отдельно. Если ввести *IDiscountCalculable*, поток расчёта можно переписать через единый проход — порядок применения будет явным и централизованным, а не размазанным по отдельным if-блокам.

## Общая рефлексия

Честно говоря, в начале я думал, что примеси — это просто «интерфейсы с методами». Технически так и есть. Но после трёх итераций на живом коде стало понятно, что дело не в синтаксисе.

Главное, что я вынес: дублирование бывает двух видов. Первый — случайное, когда копировали и забыли вынести. Второй — скрытое намерение, когда классы кажутся похожими, но предполагалось что они разойдутся. Без примесей эти два случая выглядят одинаково. С примесями — по-разному. *Promocode* не реализует *IAuditable* — это читается сразу, без объяснений.

Попутно разобрался что *ILoggable* с *Console.WriteLine* в теле — это инфраструктура в домене. Мелочь на первый взгляд, но именно такие мелочи превращают архитектуру в кашу. Интерфейс должен задавать контракт, реализацию — отдавать *ILogger* из DI.

С *IDiscountCalculable* история другая. Там не было скрытого намерения — просто формула дублировалась механически. Примесь решает две вещи сразу: убирает дублирование и централизует инвариант. Но здесь же нащупал ограничение: если правило нестабильно и инструменты могут разойтись — примесь становится клеткой. В такой ситуации нужна стратегия, а не примесь.

Третья итерация добавила ещё одно правило в копилку: *is SomeInterface* внутри бизнес-метода — сигнал что контракт неполный. Если поведение нужно всем участникам потока, оно должно быть в интерфейсе явно.

Для себя сформулировал простой признак: если вижу одинаковый метод в двух классах — сначала спрашиваю, одно ли это правило или просто похожий код. Если одно правило и оно стабильно — примесь. Если классы могут разойтись — оставляю раздельно, но фиксирую это явно. Если нужен полиморфизм — контракт должен быть полным, без проверок типов в бизнес-коде.
